# 에이전트 팀 아키텍처

> 본 문서는 **수익화 시나리오에 패턴을 붙이는 요약**이다. Harness **용어·오케스트레이션·스킬 규칙** 전체는 [harness-engineering.md](harness-engineering.md)를 본다.
>
> 본 저장소의 `../agents/` 파일은 **역할·프로토콜 스펙**이다. 런타임에서는 Cursor 서브에이전트·태스크 또는 [Harness](https://github.com/revfactory/harness)의 팀 API(`TeamCreate`, `SendMessage` 등)에 매핑할 수 있다.

## 수익화 패턴 요약 (개념도)

전체 운영은 **생성-검증(Producer-Reviewer)** 로 산출물 품질을 고정하고, 분석·빌드·마케팅 병렬 구간은 **팬아웃/팬인(Fan-out/Fan-in)** 으로 모델링한다. Harness 공식 패턴 명칭은 아래 표와 [agent-design-patterns](https://github.com/revfactory/harness/blob/main/skills/harness/references/agent-design-patterns.md)를 따른다.

```
                    [오케스트레이터: 나]
                          |
          +---------------+---------------+
          v               v               v
   [product-analyst] [fullstack-builder] [content-marketer]
          +---------------+---------------+
                          v
               [revenue-optimizer]
```

## Harness 6가지 패턴과 본 전략 매핑

| Harness 패턴 | 역할 | 본 전략에서의 사용 |
|--------------|------|-------------------|
| 파이프라인 | 순차 단계 | 입력 스크립트 → 구조화 → 인터랙션·문항 → 패키징 (아이디어 A 핵심 흐름) |
| 팬아웃/팬인 | 병렬 후 통합 | 분석·개발·콘텐츠 동시 진행 후 오케스트레이터가 통합 |
| 전문가 풀 | 입력별 전문가 선택 | 도메인(방산/항공/일반 SaaS)별 프롬프트·검수 기준 스위칭 |
| 생성-검증 | 생성 후 검증 루프 | 코드·교안·제안서의 리뷰/재시도(최대 2~3회 권장) |
| 감독자 | 동적 작업 분배 | 에이전시 일정·우선순위·클레임 기반 태스크 (아이디어 B) |
| 계층적 위임 | 상위→하위 분해 | “MVP 총괄 → 기능 단위 위임” (아이디어 A SaaS 로드맵) |

## 복합 패턴 (실전 기본형)

| 복합 | 구성 | 적용 |
|------|------|------|
| 팬아웃 + 생성-검증 | 병렬 산출 → 각각 또는 통합 후 검수 | 주간 스프린트 산출물 품질 고정 |
| 파이프라인 + 팬아웃 | 순차 구간 내 병렬 | PRD 확정 후 UI/API/마케 병렬 |
| 감독자 + 전문가 풀 | 중앙이 유형 분류 후 전문가 호출 | 클라이언트 산업별 에이전시 견적 |

## 아이디어별 패턴 매칭 (1줄 근거)

| 아이디어 | 주 패턴 | 근거 |
|----------|---------|------|
| A: CBT SaaS | 계층적 위임 + 전문가 풀 | 교안 유형·보안 요건에 따라 하위 파이프라인을 선택 호출해야 함 |
| B: 에이전시 | 감독자(Supervisor) | 작업량·마감에 따라 빌더·리뷰어를 동적 배정하는 구조에 가장 근접 |
| C: 교육 플랫폼 | 팬아웃/팬인 | 강의 단원·자료·퀴즈를 병렬 제작 후 한 커리큘럼으로 합침 |

## 실행 모드 선택 (Harness 의사결정 트리 요약)

- **에이전트 간 메시지·교차 검증이 자주 필요**하면 에이전트 팀 모드를 우선 고려한다(예: 리서치, 코드 리뷰, 통합 기획).
- **독립 수집 후 한 번만 합치면 되는** 작업은 서브 에이전트로도 가능하다(비용·지연 절감).
- 본 설계의 `product-analyst` ↔ `fullstack-builder` 간 요구사항 동기화가 잦으면 **팀 통신 프로토콜**을 활성화하는 편이 유리하다.

에이전트별 정의는 [../agents/](../agents/)를 참조한다.
